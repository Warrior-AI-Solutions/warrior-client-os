## 6. THE THREE CRITICAL FAILURE MODES TO AVOID

### Failure Mode 1: The Hammer Problem â€” Data-Driven Onboarding That Ignores the User Journey

**What happened in the app:** Fact Maps correlated with high retention. The team concluded: "Everyone must have Fact Maps immediately." The result was overwhelming onboarding and terrible UX for new users. Data-driven â‰  user-centered.

**What this means for Client OS:** We will be tempted to front-load the engagement with everything we know how to build. **The rule:** No client sees the full technical stack in Week 1. They see one thing that works.

### Failure Mode 2: The Lost Documentation Problem â€” No Single Source of Truth

**What happened:** Prior onboarding documentation was destroyed/lost. The team had to rebuild from scratch. This was not a backup problem â€” it was an architecture problem. Documentation was single-source and non-versioned.

**What this means for Client OS:** The Project Site IS the living documentation â€” but it only works if we treat it as the single source of truth and discipline ourselves to update it throughout the engagement. The moment we start maintaining separate documents (Notion page here, email thread there, Slack message somewhere else), the site goes stale and loses its power.

**The rule:** Every project decision, scope change, milestone update, and architecture change happens on the Project Site first. Everything else references the site, not the reverse.
